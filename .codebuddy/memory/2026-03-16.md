## 遇到的问题及解决方案

### 问题 1：编译命令长时间卡住（看似无响应）
- **根因**：首次编译时 PlatformIO 自动下载 Arduino 框架（~50-100MB），下载无进度显示
- **验证命令**：`pio pkg list -d <project_path>`
- **解决**：等待下载完成，或提前检查依赖状态

### 问题 2：CodeBuddy 命令执行超时
- **根因**：CodeBuddy 有 60-120 秒超时保护
- **表现**：长时间命令被自动跳过
- **解决**：依赖下载完成后，编译仅需 8-10 秒，不会触发超时

### 问题 3：ledcAttach API 编译错误
- **错误信息**：`'ledcAttach' was not declared in this scope`
- **根因**：Arduino Core 版本 API 差异
- **解决**：使用兼容 API：`ledcSetup()` + `ledcAttachPin()`

## 烧录问题深度总结与 SKILL 改进

### 问题复盘：连续烧录场景下的串口占用

**问题发生过程**：
1. 第一个任务（LED呼吸灯）成功烧录并运行
2. 用户提出第二个任务（按键控制LED）
3. 编译成功后，执行烧录时出现 `PermissionError(13, '拒绝访问')`
4. 多次重试 `pio run -t upload` 均失败
5. 最终使用 `esptool.py` 直接调用成功烧录

**根因分析**：
| 因素 | 说明 |
|------|------|
| 程序运行中 | 上一个程序（呼吸灯）持续运行，可能使用串口输出日志 |
| 设备重启 | ESP32 每次烧录后会重启，串口短暂锁定 1-3 秒 |
| Windows 串口机制 | Windows 对串口的独占访问机制导致无法同时打开 |

**命令执行失败记录**：
```
1. pio run -t upload                    → Error 2 (无详细输出)
2. pio run -t upload --upload-port COM6 → Error 2
3. python esptool.py ... --port COM6    → PermissionError(13)
4. 等待几秒后再次执行 esptool           → 成功
```

### 完成时间

### 传感器规格
| 参数 | 值 |
|-----|-----|
| 名称 | 超声波测距传感器 |
| 测量范围 | 3cm ~ 350cm |
| 通信方式 | 单线脉冲（触发+回波） |
| 推荐接口 | 接口1/3/4/6（数字接口） |

### 接口 GPIO 映射
| 接口编号 | 信号引脚(PIN1) |
|---------|---------------|
| 接口 1 | GPIO5 |
| 接口 3 | GPIO2 |
| 接口 4 | GPIO7 |
| 接口 6 | GPIO3 |

### ⚠️ 重要注意事项
- **不建议连接接口5（D5）**：会与 I2C 引脚（GPIO39/40）冲突
- **与 LED 矩阵配合时必须使用非阻塞方式**（millis() 定时）
- 测量间隔建议 100ms 以上

### 完成的示例项目

#### 1. `ultrasonic_test/` - 基础测距
- 接口：Grove1 (GPIO5)
- 功能：串口输出距离值

#### 2. `ultrasonic_led_map/` - 距离映射LED
- 接口：Grove6 (GPIO3)
- 功能：距离越近，点亮的LED越多
- 映射规则：
  - 0-10cm → 9颗LED
  - 10-20cm → 8颗LED
  - ...每10cm少亮1颗
  - >100cm → 全灭
